## Escopo e método

A base ficou fixada em `bef2e868f157659b403fe1303ee121fb69fec9e6`. Para cada head de origin, `atrás/à frente` vem de `git rev-list --left-right --count BASE...SHA`; `patches +/-` vem de `git cherry BASE SHA`, onde `+` é patch sem equivalente na base e `-` é patch equivalente já encontrado na base. Arquivos exclusivos são o diff desde o merge-base (`BASE...SHA`). “Integrada” exige que o tip seja ancestral da base ou que não haja patch `+`. A classificação nunca autoriza exclusão.

## Grupos e relações de equivalência

1. **Integradas/ancestrais de main:** `main`, `2.0.x`, `pharr117/base-2.5.2`, `release/v2.4.x` e `release/v2.5.x`. Não há patch exclusivo a reaplicar.
2. **Sem duplicatas exatas:** nenhuma das 79 heads compartilha tip SHA ou tree SHA. “Mesmo tema” abaixo não significa branch duplicada.
3. **Linha de dependências:** `colin/bump-ibc-go-v6` é ancestral de `colin/bump-sdk-v47`; a segunda acrescenta o bump do SDK v0.47. `dan/update-go-sdk` é uma tentativa paralela antiga. As heads Dependabot de SDK/IBC/CometBFT são patches unitários sobre um merge-base mais recente.
4. **Família Client ICQ/ICA:** `andrew/client_icq_stride_v3` é ancestral de `andrew/client_icq`; `andrew/fix_timeout_on_close` também é ancestral de `andrew/client_icq`. `andrew/client_icq_ica_fix` divergiu, mas 28 patches de `client_icq_stride_v3` têm equivalentes nela. Revisar como uma família, não somar commits cegamente.
5. **Penumbra:** `penumbra_buf` é ancestral de `penumbra`; a segunda contém a integração funcional maior.
6. **Gordian:** `reece/gordian` e `reece/gordian-consensus` divergem e não têm patches equivalentes detectados; a branch `gordian` contém commits marcados “REVERT ME/REMOVE ME”, enquanto `gordian-consensus` é o material de interface mais focado.
7. **Handshakes/flush:** `andrew/connection_channel_handshakes`, `andrew/path_processor_automatic_handshakes_poc`, `andrew/flush_channels_connections` e `andrew/justin-collab-close-channels` são experimentos distintos e não devem ser tratados como uma sequência linear.
8. **Eventos/processador:** `andrew/attribute_parsing_old`, `andrew/event_bleed`, `andrew/event_processor`, `andrew/fix_ica_handshake`, `andrew/cosmos_message_handlers_2`, `andrew/cosmoschainprocessor_big_blocks` e `andrew/websocket` preservam correções/experimentos não equivalentes em parsing, acumulação e ingestão de blocos.

## Decisões recomendadas

1. Criar uma branch de manutenção nova a partir de `bef2e868…`; não promover nenhuma branch histórica inteira.
2. Tratar os upgrades SDK/CometBFT/IBC como uma matriz coordenada e moderna. As branches v6/v8/v9 são evidência de migração e regressões, não destino de versão.
3. Portar primeiro testes/comportamentos de alto valor: parsing multi-evento, ICQ/ICA, flush de handshakes, recovery de light client, 08-wasm, isolamento de falhas e abstração de consenso.
4. Para cada branch prioritária, extrair commits `+` e arquivos listados, escrever teste contra a base e só então reimplementar/cherry-pick seletivamente.
5. Não excluir nenhuma ref com base neste relatório. `docs-site` exige auditoria própria por não compartilhar história com `main`.

## Limitações

- Não houve fetch para manter a auditoria somente leitura. A validação mostrou que todos os 79 SHAs live de origin já existem no banco local e que as refs `origin/*` correspondem ao snapshot live, portanto os cálculos Git são reproduzíveis neste checkout.
- Heads de PRs vindos de forks externos foram analisados via metadados live do GitHub; apenas os que também aparecem em origin têm objetos locais incluídos na tabela Git.
- `git cherry` mede equivalência de patch, não equivalência semântica. Branches podem implementar a mesma ideia de maneiras diferentes.
