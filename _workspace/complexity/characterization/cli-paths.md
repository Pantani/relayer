- Observable contracts:
  - Cobra Use, aliases, argument cardinality, and all JSON/YAML/File/Filter/ID/Overwrite/Testnet defaults are frozen. List ignores errors from absent JSON/YAML getters and behaves as plain output. Update's Filter Rule and Filter Channels use the `blank` sentinel while every path-end flag defaults to empty. Fetch accepts zero or one chain.
  - List rejects simultaneous JSON/YAML before path-status queries. Plain output retains index zero, fixed columns, glyphs, and chain-pair formatting. JSON and YAML retain exact Path map envelopes and newline behavior.
  - Show resolves path existence, then both configured chains, before its ignored format getters and status query. Missing path and missing-chain errors remain exact. Plain, compact JSON, and YAML retain exact PathWithStatus field names, status values, indentation, glyphs, and trailing newlines.
  - Add resolves both positional chain IDs before reading `--file`. Missing-chain errors therefore precede flag-access errors. In File mode, the path ends inside the file win over the positional chain IDs; a reversed but configured path is validated, added under the positional path name, and persisted silently. Invalid identifier syntax in an added file fails validation before mutation or save.
  - Update loads the named path through `MustGet`, so a missing name panics after config reload. Invalid Filter Rule returns before later field updates. A no-flag update returns `at least one flag must be provided`. All eight fields update and persist in their existing order; Filter Channels split on commas, explicit empty Filter Rule clears it, and explicit empty Filter Channels produces nil in memory but an empty slice after YAML reload.
  - Update does not verify that new chain IDs are configured and does not validate Classic client/connection identifier syntax; such values save successfully. Its post-operation structural validation can still fail after mutating runtime memory. For IBC v2, adding a source connection ID mutates memory, is rejected by `validateConfig`, and leaves the on-disk V2 path unchanged.
  - Fetch checks an explicitly requested chain against current in-memory config before taking the config lock or touching the network. Existing paths skip with the exact stderr line and no request unless Overwrite is set. Overwrite refetches the path and clears its filter. Testnet changes the Contents API directory to `testnets/_IBC`; successful fetch maps registry chain names to configured chain IDs, writes the exact client/connection IDs, emits `added`, and persists.
  - GitHub v43 `DownloadContents` performs two requests per success: directory listing followed by `download_url`. Retrieval errors are nonfatal and emit `failure retrieving` before later combinations continue. Fatal JSON unmarshal stops the batch immediately: a prior successful addition and `added` line remain in memory/stderr, no third request runs, and the disk remains unchanged because the locking operation never reaches save.
  - Pair enumeration is backed by a Go map and therefore has no stable lexical order. Tests preserve the invariant that stderr lines follow the actual processing/download order without asserting a fixed first pair. In a three-chain batch, one directory lookup failure is followed by two successful downloads/additions and only successful paths are persisted.
- Network seams: tests replace `http.DefaultTransport` only during sequential test scope and restore it with cleanup. Fixtures implement the real two-stage GitHub Contents API contract using a directory JSON array and local `https://download.test/<pair>.json` responses. No request reaches the public network.
