- Observable contracts:
  - The RPC inventory is a new ordered slice containing the configured primary followed by every backup exactly as configured. Empty strings and duplicates are retained and counted.
  - Cosmos always logs `Available RPC clients` and enters its ticker loop, even with no backups. Penumbra returns immediately when the inventory has one entry and logs `No backup RPCs defined` only when a logger exists.
  - With more than one RPC, both log `Available RPC clients` with `chain` and `count` fields before reading the context. A nil context then panics. Cosmos also panics on a nil logger before observing cancellation; Penumbra is nil-logger safe only in its no-backup early return.
  - A healthy status tick performs no reconnect attempt and leaves the consensus client, light provider, primary config, and backup config unchanged.
  - A failed status tick logs disconnection, then tries primary, duplicate primary, and backup in that exact order. Attempts are consecutive inside one tick; there is no per-endpoint backoff. The next health opportunity is the hard-coded ten-second ticker.
  - Successful rotation installs the backup consensus client and light provider, but does not update `PCfg.RPCAddr` or `PCfg.BackupRPCAddrs`.
  - When every endpoint fails, all configured endpoints are attempted once and the terminal error is logged. Because `setRpcClient` assigns before its status check, the consensus client is partially mutated to the last failed endpoint; the light provider and provider config remain unchanged.
  - Cosmos reconnect failures are logged at Error. Penumbra reconnect failures are logged at Debug. Both log disconnection and the all-endpoints-failed terminal event at Error, with chain/RPC/error fields supplied by production.
  - Context cancellation before the first tick or after a healthy/success/all-fail tick terminates the function and its goroutine. Test-owned completion channels prove termination within one second after cancellation.
  - Production does not close or emit a result channel. It selects only on `ctx.Done()` and `ticker.C`. The ticker is not explicitly stopped on return.

- Gaps/blockers:
  - The ten-second interval is constructed directly with `time.NewTicker`, so deterministic tick coverage must wait one real interval. No production seam was introduced because production is read-only during characterization.
  - The local servers return immediately; cancellation while a reconnect status request itself is blocked is not separately exercised. `setRpcClient` performs reconnect status checks with `context.Background()`, an existing behavior outside the target function's direct control.
  - Tests prove the monitoring goroutine exits, but cannot directly observe whether the unstopped ticker's runtime resource is reclaimed before garbage collection.
  - No blocker prevents a structural refactor. It must preserve the provider differences and the observed all-fail partial consensus-client mutation unless that behavior is changed in a separate approved scope.
