- Observable contracts:
  - Every direct call first writes `channelConnections[ChannelID] = ConnID`, including empty identifiers and overwrites.
  - `ChannelInfoChannelKey` supplies the exact key for channel state and handshake retention. The retained value is the complete `ChannelInfo` copy, including height, connection identifiers, order, and version.
  - `channel_open_init` inserts a closed state under the init key unless any existing full key has the same `MsgInitKey`. An existing full state is preserved without adding a duplicate init state.
  - `channel_open_try` writes the full key closed. `channel_open_ack` and `channel_open_confirm` write it open. These non-init events remove the init key.
  - `channel_close` and `channel_close_confirm` close the full key only when iteration finds an existing state with the same local port and channel. An init key itself satisfies that match. With no matching state, unrelated states remain untouched and no full state is inserted.
  - `channel_close_init` is not handled by a dedicated switch case. Like an unknown event, it removes the init key, retains the message under its event string, and emits the normal observed-message log without adding channel state.
  - Handshake retention creates the per-event inner map when absent. Repeating the same event and key overwrites the prior `ChannelInfo` and remains one entry. Different event strings remain independent map buckets. Go map iteration order is not made observable or stabilized.
  - `ChannelStateCache.SetOpen` preserves an existing non-NONE order when an overwrite supplies `NONE`, while the retained message still contains the new `NONE` value.
  - Zero-value `ChannelInfo` is valid and mutates the empty-string connection entry, zero channel key, state cache, handshake cache, and logs. A nil untyped `IbcMessage.Info` is ignored by dispatch; a typed-nil `*chains.ChannelInfo` panics during dereference before channel effects.
  - Nil `channelConnections` panics before any other effect. Nil `channelStateCache` panics after the connection-map write. Nil `ChannelHandshake` panics after connection/state/init-key effects but before the final debug log.
  - Cosmos emits `Successfully created new channel` at Info for ack/confirm before init-key deletion and handshake retention. Penumbra has no equivalent Info log. Therefore a nil logger on ack panics early in Cosmos, leaving the init state and no retained handshake, while Penumbra deletes init, retains the handshake, and panics only at the final debug log.
  - With a nil handshake cache on ack, Cosmos has already emitted its Info log before the panic; Penumbra has emitted no log. Both have already opened the full state and removed init.
  - Both providers emit `Observed IBC message` at Debug after successful retention with chain name, chain ID, event, local/counterparty channel and port, and connection fields. Cosmos' additional Info log includes channel, connection, and port fields.
  - The handler does not inspect path processors, perform path filtering, return errors, call providers, access metrics, or perform network I/O. Tests use nil path processors and nil metrics to freeze that absence of side effects.

- Gaps/blockers:
  - Concurrent calls are not characterized because the production maps are unsynchronized and the chain processor invokes this handler serially. The race gate validates the characterization suite itself, not unsupported concurrent map mutation.
